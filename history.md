更新时间：2026-07-05

当前项目地址：`https://agentjz.github.io/ohmyflight/`

用途：给后续窗口继续理解 ohmyflight。它不是 `spec/`，不是 README，也不是产品宣传。`spec/` 只写当前工具怎么用；这份文件写历史：业务怎么长出来，哪些路试过又删掉，人和 AI 怎么从反复误解走到现在的协作方式。

## 一句话

ohmyflight 从一开始就不是炫技项目。

它的价值一直是实用：把飞行部真实工作里烦、碎、容易错、重复做的事，变成能上传、能核对、能导出、能复查的工具。

技术栈只是承载方式。真正主线有两条：

```txt
业务史：真实工作 -> 工具探索 -> 删除无效工具 -> 培训皇帝 / 审计之王等核心工作台
协作史：口头纠偏 -> 反复误解 -> spec / test / skill / AGENTS 固化事实 -> 新窗口也能接上
```

- ohmyflight 的前身是 777F / 777发。

- 没有证据的内容不写成事实；项目所有者明确提供的背景作为 owner 事实记录。

# 第一部分：业务史

## 阶段 00：种子期，先把实际活变成工具

时间：早期到 2026-03

owner 事实：

777F / 777发是 ohmyflight 的前身。第一版来自 Void Editor，早期也有 Kiro 参与。

当前可见 Kiro 证据：

- 项目被描述为“飞行部工具箱”。
- 目标是本地、离线、纯前端。
- 工具围绕 PDF、Word、Excel、Markdown、图片、OCR 和飞行业务表格展开。
- 页面形态偏传统：HTML、Bootstrap、iframe、本地 `libs/`。

当时真正要解决的问题不是“做一个网站”，而是把工作里重复的活变成工具：

- 表格太多，人工筛选慢。
- 姓名、员工号、日期、培训项目容易看错。
- PDF、Word、Excel 处理经常重复。
- 有些工作不需要服务器，只需要本地把口径算清楚。

走对的地方：

- 先解决真实工作，不先追技术先进。
- 数据尽量留在本地，不上传。
- 一个工具解决一个具体问题。
- 工具可以粗糙，但必须能产出结果。

埋下的问题：

- 工具越来越多后，入口会乱。
- 每个工具自己写上传、解析、导出，重复会越来越多。
- 业务规则只在代码里，后续容易忘。
- 没有测试时，规则改动很容易把旧场景改坏。

最后留下的判断：

ohmyflight 的项目气质从一开始就是“把活干完”。这个判断到现在都没有偏移。

- 换季学习查询。
- 危险品、英语、汉语考试培训规则。

这些工具不是抽象需求。它们来自实际工作里会反复出现的“人肉劳动”。人工也能做，但慢、烦、容易错。

走对的地方：

- 先自动化最烦的重复劳动。

- 工具入口和代码结构还不够稳。
- 训练、锁班、飞行经历这些重要工具逐渐暴露：只“能跑”不够，必须能解释结果。

最后留下的东西：

777F 阶段证明了项目有真实价值。后面迁移、重构、测试、spec，都是为了让这些真实价值长期站住。

## 阶段 02：工具箱膨胀，什么都想放进去

时间：2026-04 上旬

当时出现或讨论过的工具方向：

- PDF 工具。
- Word 模板填充。
- Excel 批处理。
- Markdown / 文档查看。
- 图片处理。
- OCR。
- 文件批量工具。
- 账单 / 甘特 / 航班 / 名单类工具。

这一阶段有一半是对的，一半是危险的。

对的地方：

飞行部工作确实会遇到各种文件处理和表格处理。PDF、Word、Excel、图片、本地模板这些能力不是装饰，它们是办公工具箱的底座。

危险的地方：

不是每个能做的工具都值得长期维护。工具太多会让首页和维护成本失控。通用文件工具如果没有明确业务场景，很容易变成“做了但不常用”。

最后留下的判断：

ohmyflight 可以有通用办公工具，但必须服务真实工作。工具不是越多越强，留下来的必须有使用场景。

## 阶段 03：平台化试探，后来证明不是主线

时间：2026-04 中旬

当时试过的方向：

- 通用 workflow 平台。
- 类似 n8n / Coze 的节点编排。
- React 拖拽节点。
- 插件接口。
- 工作流工具集合。

为什么当时想做：

工具越来越多后，很自然会想到“能不能把所有工具变成节点，用工作流串起来”。它看起来更高级，也似乎能解决复用问题。

后来证明为什么不对：

- 用户真正要的是“上传这份表，得到这个结果”，不是理解节点系统。
- 通用平台会把简单工具复杂化。
- 很多业务问题不是流程编排问题，而是业务口径问题。
- 平台一旦做起来，会吞掉培训、锁班、账单这些真正重要的业务规则。
- React / 构建 / MIME / 本地服务问题当时也制造了很多和业务无关的返工。

最后留下的实践：

ohmyflight 不做大一统工作流平台。它选择“独立业务工具 + 统一入口 + 统一 spec + 统一测试”的路线。

这条探索路线很重要：

它说明项目不是没想过平台化，而是试过以后发现主线不在这里。ohmyflight 要解决的是具体工作，不是展示一个通用系统。

## 阶段 04：从 777F 迁移到 ohmyflight，拒绝假迁移

时间：2026-04-15 到 2026-04-16

关键提交：

- `369ff49 2026-04-15 create spec`
- `48b397d 2026-04-15 已迁移，停止维护`
- `67d5521 2026-04-16 typescript rewrite,tool,workflow`

当时的核心问题：

777F 能做事，但长期维护开始吃力。迁移到 ohmyflight 不是换名字，而是把项目从“工具堆”变成“可维护的业务工具仓库”。

当时明确退出的路线：

- 旧文件夹继续被新代码依赖。
- TypeScript 只是包一层旧 JS。
- 只让构建通过，不整理业务事实。
- 把旧工具全部无脑搬过来。

这里有一个重要协作经验：

AI 很容易把“迁移”理解成目录和构建迁移。项目所有者真正要的是“第一方事实只剩 TS，业务逻辑可维护，错误尽量在类型和测试阶段暴露”。所以当时才会反复追问：

```txt
这是不是挂羊头卖狗肉？
用 TS 重构的意义在哪里？
```

最后证明正确的路：

- 建立 `src/`、`public/`、`spec/`、`tests/`。
- 工具入口集中管理。
- spec 记录当前用户侧规则。
- 测试保护关键业务口径。
- 旧项目停止维护，新项目接住主线。

这阶段留下的核心原则：

迁移不能是假迁移。真正迁移必须让业务规则、文件结构、入口、测试、文档都进入新主干。

培训皇帝为什么是核心实践：

它不是 Excel 筛选器。它处理的是一整本培训总表，里面有人员信息、CRM、多个项目 sheet、有效期、排班记录、录入状态、取消备注、教员、窗口期、年度压力、补训、导出。

任何一个字段理解错，结果就会错。

### 早期路线：到期清单、预排、培训计划核对

最早的培训工具更像几个入口：

- 到期清单查询。
- 预排班设置。
- 培训计划核对。
- 有效期更新。

这些入口当时合理，因为问题还没有完全收束。后来它们没有作为当前可见入口保留。当前培训皇帝只保留：

- 排班总览。
- 有效期更新。

为什么收束是对的：

用户真正需要的是“当前谁要处理、为什么要处理、已有安排是否覆盖、能不能导出”，不是一堆分散入口。入口越多，越容易让同一套业务规则分裂。

### 第一轮硬口径：有效期内已安排就是 OK

早期容易犯的错：

如果某人本月没安排，就认为有问题。

后来证明不对：

如果已经提前安排，并且安排在有效窗口内，就应该 OK。培训不是看“本月有没有安排”，而是看“已有安排能不能覆盖当前有效期风险”。

这个修正把工具从简单筛选推进到真正业务判断。

### 第二轮硬口径：项目规则不一样

培训项目不是统一规则：

- 应急训练有基准月。
- 危险品是 3 个月窗口，但失效前一日截止。
- 英语能力、汉语能力是 3 个月窗口。
- 航空安保、TSA、疲劳管理、飞行作风按最新日期。
- TSA 和航空安保后来被明确拆成两个独立有效期项目。

走过的探索：

把项目当成同一种有效期规则。

最后证明正确的路：

每个项目必须有自己的规则。规则写进 `training-rule-spec.md`，测试覆盖典型样例。培训皇帝必须按规则判断，而不是按项目名大概猜。

### 第三轮硬口径：航空安保和 TSA 不能混

一开始容易把 TSA 和航空安保看成一组相关训练。

后来明确：

- `航空安保` sheet 只证明做了航空安保。
- `TSA` sheet 只证明做了 TSA。
- 同一天同时做了航空安保和 TSA，也应该两个项目 sheet 各记录一条。
- 航空安保有效期 24 日历月，TSA 12 日历月。

这说明培训皇帝不能靠“看起来类似”合并规则。业务上是两个项目，代码里也必须是两个项目。

### 第四轮硬口径：`培训信息是否录入` 不是覆盖判断

这个字段非常容易误用。

最后规则是：

- 排班总览只判断是否存在有效安排。
- 只要项目 sheet 中已排且备注没有取消，就参与覆盖判断。
- `培训信息是否录入` 是人工录入状态字段，不参与排班覆盖判断。
- `机器看 = Y` 只用于有效期更新。

如果把 `培训信息是否录入` 当成覆盖判断，未录入但已安排的计划会被错误忽略。业务上，排班总览要解决的是“有没有安排覆盖风险”，不是“后台是否录入完成”。

### 第五轮硬口径：取消备注

备注包含 `取消` 的记录必须特殊处理。

最后规则是：

- 未录入记录备注取消：这条计划无效，不参与覆盖判断。
- 已录入或 `机器看 = Y` 但备注取消：数据矛盾，应异常提示或跳过。

这个规则看起来小，但很容易造成误判。培训工具不能只看日期，还要看备注语义。

### 第六轮硬口径：CRM 是年度核对，不是有效期项目

CRM 曾经容易被混进培训有效期逻辑。

最后规则是：

- CRM 是独立年度核对模块。
- 只读取精确名为 `CRM` 的 sheet。
- `CRM2025(不参与统计)` 这类历史 sheet 不参与。
- 指定年份内有培训开始日期或结束日期，且备注不包含取消，判定已参加。
- `培训信息是否录入` 不参与 CRM 参加判断。
- 人员信息表里 `是否运行 = 否` 的人员也进入应参加名单。
- CRM 教员不进入未参加名单。

CRM 不是“另一个培训项目”。它是年度参加核对。它只分已参加和未参加，不产生推荐排、必须排、窗口期、覆盖判断或有效期更新。

### 第七轮硬口径：健康检查不是第二套规则

培训总表很复杂，用户导入后必须知道程序怎么理解这本 Excel。

因此增加 Excel 健康检查：

- 说明识别了哪些 sheet。
- 暴露严重、警告、提示。
- 复用现有解析器、取消备注策略和 CRM 年度核对逻辑。
- 不维护第二套业务规则。

健康检查不是装饰。它是防止“工具默默算错”的保险。

### 第八轮硬口径：工具呈现事实，不替人乱判断

培训皇帝不是自动替人判断一切。

它围绕培训规则、窗口期、到期压力、已排覆盖、取消备注这些明确口径呈现事实。无窗口项目提前 2 个月推荐排、到期区间必须排，这属于必要业务判断。超出规则的“模型脑补判断”必须避免。

这条线后来变成协作原则：

```txt
机器和代码做确定事实。
人确认业务口径。
AI 不凭感觉替业务下判断。
```

### 培训皇帝最后变成了什么

当前培训皇帝面向南货航新总表结构，直接导入一整本培训总表。核心判断链是：

```txt
人员信息表当前有效期 -> 项目 sheet 有效排班记录 -> 培训规则 -> 备注取消特判
```

它保留两个可见入口：

- 排班总览。
- 有效期更新。

它不再保留独立的：

- 到期清单查询。
- 预排班设置。
- 培训计划核对入口。

这个收束是实践结果，不是功能缩水。分散入口曾经存在，但最后证明同一套规则应该在一个主工作台里闭环。

## 阶段 06：删除和收束无效工具，成熟不是越堆越多

时间：2026-05 到 2026-06

曾经出现后被删除或收束的方向：

- `training-tools-suite`
- `super-training-test`
- `bill-gantt`
- `file-batch-tool`
- `flight-filter`
- `health-poster`
- `seasonal-learning-check-test`
- `tool-mode-toggle`
- `workflows-data`

这些不等于当时做错。

很多工具在当时有探索价值。问题是：一个项目长期维护时，不能把每个探索都留在首页和主干里。

### `training-tools-suite` 为什么退出

它曾经承载规则参考、预排、冲突检查、有效期更新等多个入口。

问题是：

- 入口太分散。
- 同一套培训规则容易在多个页面重复。
- 用户要的是统一判断链，不是到处找入口。

最后它被培训工作台 / 培训皇帝吸收。

### `super-training-test` 为什么改名和收束

`super-training-test` 是培训皇帝的前身之一。

问题是：

- 名字像测试，不像正式工具。
- 逻辑一度过度合并。
- 到期清单、计划核对、预排、有效期更新这些能力需要重新梳理。

最后变成 `training-workbench`，用户侧叫培训皇帝。

### `bill-gantt`、`file-batch-tool`、`health-poster` 等为什么退出

它们更像通用能力、阶段性工具或边缘工具。

问题是：

- 和当前核心业务的绑定不够强。
- 长期维护优先级不如培训、账单核对、名单、锁班、审计这些硬场景。
- 留太多会让首页越来越像杂货铺。

最后它们退出主线。

这一阶段最后留下的判断：

删除不是退步。删除是项目变成熟的一部分。ohmyflight 不追求“我做过多少工具”，而追求“留下来的工具是不是能长期解决真实工作”。

## 阶段 07：锁班、人员结构、OA，外部系统自动化必须能核验

时间：2026-05 下旬

关键提交：

## 阶段 10：维护性收束，技术原则服务业务稳定

时间：2026-06 下旬到 2026-07-05

- `4f6e674 2026-07-05 拆分重点工具入口职责并补强测试`

为什么要重构：

不是为了“代码优雅”，而是因为业务规则越来越多。如果页面、状态、解析、导出、规则、DOM 全塞在一起，每次修一个规则都可能撞坏另一个工具。

最后形成的技术原则：

- 业务逻辑和页面渲染分开。
- 解析、统计、规则判断放可测试模块。
- 页面只负责上传、展示、导出、状态提示。
- spec、代码、测试必须讲同一个当前事实。
- 超过 300 行触发职责审查，但不为了拆而拆。
- Python 工具和浏览器工具边界清楚。
- GitHub Pages 部署保持简单稳定。

这部分是技术史，但不是主线。

技术重构的目的，是让培训、审计、锁班、账单这些业务工具不被未来改坏。

# 第二部分：协作史

## 协作阶段 00：早期靠口头纠偏，浪费很多时间

早期人和 AI 的协作最大问题不是“模型不会写代码”，而是没有共享事实。

用户说的是业务现象：

- 这个人这个项目在该月有任意一条记录就算。
- 提前安排且在允许范围内就算 OK。
- 导出只是给我看，不是二次导入。
- 插名字、标红、带有效期就够。
- 工具只是呈现事实，不要替我判断一切。

AI 很容易听成技术任务：

- 顺手建索引。
- 顺手抽象公共模块。
- 做一个更通用的流程。
- 做更多入口。
- 解释一个看起来“工程上更干净”的方案。

这就造成反复纠偏。

典型浪费点：

- AI 没先看仓库就开始判断，所以用户会说“你是不是先 research 一下我的仓库再说话呢？”
- AI 把迁移理解成目录结构变化，用户要的是可维护的业务事实迁移。
- AI 把业务规则理解成技术优化，用户要的是先把口径钉死。
- AI 想顺手做性能或结构优化，用户要的是如无必要勿增实体。
- AI 试图用通用平台、通用抽象、通用工具覆盖具体业务，最后又被拉回真实工作流。

最后学到的东西：

业务项目里，先听懂“这个活是怎么干的”，再讨论代码怎么写。

## 协作阶段 01：强约束提示词出现，因为普通对话接不住复杂任务

当项目变大后，普通一句“帮我改一下”已经不够。

用户开始写很强的任务提示：

- 一次性完成。
- 先 research。
- 不要半成品。
- 不要做一点停一点。
- 先写文档和验收标准。
- TDD。
- spec、代码、测试同步。
- 不要向后兼容旧垃圾。
- 不要挂羊头卖狗肉。

这些话不是情绪，它们来自反复校准。

强约束解决的问题：

- AI 容易只做局部补丁。
- AI 容易不验证。
- AI 容易用解释代替完成。
- AI 容易把历史旧目录当成兼容对象。
- AI 容易把用户话术直接写进代码，而不是理解业务事实。

强约束的副作用：

它能逼任务完成，但如果没有仓库规则承接，每个新窗口都要重新骂一遍、重新解释一遍、重新纠偏一遍。

所以后面必须把这些经验写进项目，而不是永远靠口头。

## 协作阶段 02：spec 开始承接业务事实

`spec/` 的作用不是写漂亮文档。

它是为了防止 AI 和人对业务事实各说各话。

在培训皇帝里，spec 解决了这些问题：

- 什么项目有什么有效期规则。
- 什么是基准月。
- 什么是 3 个月窗口。
- 危险品为什么是失效前一日截止。
- 航空安保和 TSA 为什么分开。
- CRM 为什么是年度核对，不是有效期项目。
- `培训信息是否录入` 不参与排班覆盖判断。
- `机器看 = Y` 只用于有效期更新。

在其它工具里，spec 解决了这些问题：

- 工具做什么，不做什么。
- 输入是什么。
- 输出是什么。
- 用户应该怎么核对。
- 哪些异常要暴露。

spec 的正确边界：

- spec 写当前用户侧功能事实。
- spec 不写历史。
- spec 不写未来计划。
- spec 不写实现过程。

history 的作用：

- history 写历史。
- history 解释为什么某些规则存在。
- history 记录哪些路线验证后退出。
- history 帮新窗口理解项目演进。

这两个分开以后，协作清楚很多。

## 协作阶段 03：测试从“证明能跑”变成“保护业务口径”

早期没有测试时，AI 很容易改坏规则还不知道。

后面逐渐明确：

- 修改业务规则，必须补测试。
- 修改日期规则，必须补测试。
- 修改导出结构，必须补测试。
- 修改解析规则，必须补测试。
- 曾经错过的场景，必须变成回归测试。

测试的目的不是保护目录形状，也不是证明 UI 文案没变。

测试要保护真实产品行为：

- 培训有效期规则。
- CRM 参加判断。
- TSA / 航空安保拆分。
- 酒店账单匹配。
- Word 批量导入导出。
- PDF 加水印位置规则。
- 人员结构闭合统计。
- 锁班结果匹配。

这里也有过协作经验：

AI 有时会写“某段旧代码不得出现”这类负向测试。这种测试看起来硬，实际容易保护口号。后来规则明确：除非对应真实业务可观察回归，否则不写这种负向测试。

最后留下的实践：

测试不是给 AI 刷绿的。测试是给业务规则守门的。

## 协作阶段 04：AGENTS.md 把沟通规矩写成仓库入口

后来 `AGENTS.md` 成为仓库最高约束。

它解决的是新窗口最容易犯的错：

- 不读 UTF-8，误判中文文件乱码。
- PowerShell 里直接用 `npm` / `npx`，踩执行策略问题。
- here-string 里的中文被编码污染，还以为是代码错。
- 不区分事实、推测、计划、完成。
- 没有原始输入就猜业务规则。
- 不看中间值，只看最终现象。
- 修改业务不更新 spec 和测试。
- 遇到脏工作区误删用户改动。

AGENTS.md 的核心不是“约束多”，而是把过去反复浪费时间的环节提前固化。

最重要的几条：

- 事实高于一切。
- 事实不清楚先排查。
- 不能猜业务规则。
- 用户给了原始输入，优先用原始输入复现。
- 表格、文本、日期、姓名问题，要看中间值。
- 代码、测试、文档必须同步。
- 文档只写当前事实，不写历史过程。

这些规则让协作从“每次靠用户提醒”，变成“模型接手就知道边界”。

## 协作阶段 05：skills 把不同工作流拆开

随着项目变复杂，一个总规约不够了。

于是出现项目级 skills：

- `ohmyflight-dev`：新增或修改工具的一般流程。
- `training-workbench`：培训皇帝专属规则。
- `ui-clarity`：页面 UI 清晰度。
- `playwright-probe`：内网页面探测和临时自动化。
- `docx-report`：Word 文档读取、填写、生成。
- `interview-lock-list`：面试名单转锁班导入模板。
- `plan`：中大型任务的单文件执行合同。

skills 解决的问题：

不同任务的成功标准不一样。

培训任务要先看培训规则。UI 任务要看页面清晰度。Playwright 任务要区分输入解析、DOM、提交值。docx 任务要看 Word 表格结构。大重构要有 plan。

如果都靠一个大提示词，模型会混。

skills 的价值：

它把“这类任务应该怎么做”从对话里移到仓库里。新窗口不需要重新问一遍，也不需要靠记忆猜。

## 协作阶段 06：plan.md 用来管大任务，不让任务边做边飘

当任务变成跨模块重构、工具下线、测试整理、生产级验收时，口头计划不够。

plan.md 的作用：

- 写清需求。
- 写清当前事实。
- 写清验证用例。
- 写清目标。
- 写清不做范围。
- 写清设计。
- 写清实施任务。
- 写清验证计划。
- 写清收口记录。

这不是形式主义。

它解决的是 AI 最常见的问题：

- 做着做着忘了最初目标。
- 修了局部，没收全局。
- 代码改了，spec 没改。
- 测试过了，但不是用户真实路径。
- 任务做到一半停下来让用户补洞。

plan.md 后来变成大任务的合同。它不是为了拖慢开发，而是为了防止大任务失控。

## 协作阶段 07：history.md 承接长期上下文

现在写 history，是因为对话已经太长，新窗口不可能天然知道所有历史。

但 history 不能写成干净架构图。

这个项目的历史本来就是：

- 做了。
- 发现不对。
- 删了。
- 又做。
- 又收束。
- 规则被一次次钉死。
- 协作方式也被一次次改造。

history 的正确作用：

- 让新窗口知道项目从哪里来。
- 让新窗口知道哪些路线试过后退出。
- 让新窗口知道哪些规则来自长期实践。
- 让新窗口知道用户为什么讨厌半成品、假迁移、无意义抽象。
- 让新窗口知道 spec 是当前事实，history 是历史证据。

history 不应该做的事：

- 不替代 spec。
- 不替代 README。
- 不写成宣传稿。
- 不把旧功能当当前能力。
- 不把没有证据的内容写成事实。

## 最重要的业务实践经验

- 工具不是越多越好。
- 培训规则不能靠直觉。
- 字段名相似不代表业务意义相同。
- 已安排、已录入、机器看，是三件事。
- CRM 不是有效期项目。
- TSA 和航空安保不能混。
- 取消备注不是普通备注。
- 自动化必须能复查。
- 删除无效工具是成熟，不是倒退。
- 技术重构必须服务业务稳定。

## 最重要的协作实践经验

- AI 先 research，再说话。
- 先用非技术语言对齐业务。
- 用户说的是业务现象，不要直接翻译成技术方案。
- 不要顺手优化，先判断业务必要性。
- 如无必要勿增实体。
- 不要用通用平台吞掉具体业务。
- spec 写当前事实。
- history 写历史证据。
- tests 保护业务口径。
- skills 固化不同任务的工作方式。
- AGENTS.md 是新窗口入口。
- plan.md 管大任务，防止任务漂移。

## 后续新窗口接手顺序

1. 读 `AGENTS.md`。
2. 按任务读 `.agents/skills/.../SKILL.md`。
3. 读相关 `spec/app/...`。
4. 看同类成熟工具源码。
5. 看相关测试。
6. 需要理解历史原因时，读本文件对应阶段。

最后记住一句：

```txt
真实业务问题 -> 清楚业务口径 -> 可复查输出 -> 再谈技术实现
```
