# Playwright Probe

在本仓库编写临时 Python Playwright 自动化脚本时使用本 skill。

## 核心原则

不要猜页面结构。先按用户给的录制流程跑通，再用少量样例确认页面事实，最后才写批量脚本。

用户给的录制脚本通常只代表“要走哪条业务路径、要查什么东西”。录制脚本里的 `fill`、点击顺序、等待方式和 locator 不一定稳定，不能直接当成成熟实现照抄。

本 skill 解决的是本项目里已经出现过的沟通成本：录制脚本被误当成最终实现、没有先看现有成熟 Python app、没有先证明页面结构、把固定示例值误当成业务全集、用户补充过滤条件后没有重新验证。

## 工作流程

1. 先读 `AGENTS.md`。
2. 查看 `public/tool/app/**/app.py` 下已有 Python 工具，沿用项目风格：
   - `sync_playwright`
   - 可见浏览器
   - 登录失败时允许人工接管
   - 每次查询前清空输入框
   - 用 `openpyxl` 读写 Excel
   - 每查完一行就实时写回
   - 失败行写明失败原因
   - 查询后尽量校验结果属于当前员工，避免读到上一人的残留结果
3. 用户给的 Playwright 录制脚本是第一事实来源。
   - 不要擅自换入口。
   - 不要在没跑通录制流程前自创选择器。
   - 录制脚本用于确认业务路径和页面目标；最终实现要结合成熟工具里的清空、等待、校验和写回方式。
   - 如果录制脚本能跑但写法脆弱，先用它到达页面，再基于真实 DOM 和成熟实现重写稳定逻辑。
4. 写正式批量脚本前，先写探测脚本。
   - 保存截图。
   - 保存主 frame 和子 frame HTML。
   - 页面结构不清楚时，保存可见文本和元素坐标。
   - 为样例人员保存简短探测结果。
   - 探测脚本只用于确认事实，不直接修改用户 Excel。
5. 用用户 Excel 里的 1 到 2 个已知样例先验证。
   - 把页面抽取结果和 Excel 现有样例值对比。
   - 样例不一致时，停止批量写入，继续排查页面结构。
   - 样例一致只能证明当前抽取规则可用，不能扩展成未经确认的新业务规则。
6. 从已确认页面结构中定义抽取规则。
   - 优先按表头和同行单元格读取，不要只靠全文匹配。
   - 批量写入前，先说清楚筛选列和读取列。
   - 常见模式：按某几列筛选行，再读取同一行的目标列。
   - 如果用户补充过滤条件，必须重新跑样例验证，再全量覆盖。
   - 示例值只用于验证，不代表只能查这些值；例如页面同一列还可能出现其他检查项。
7. 样例验证通过后，再写正式脚本。
   - 使用 `argparse`。
   - 用户已明确 Excel 文件时，把它作为默认输入。
   - 提供 `--limit` 方便先跑小样本。
   - 每个人独立查询。
   - 能校验当前结果属于当前员工时必须校验。
   - 每行查询完成后立刻写回 Excel。
8. 正式脚本先用 `--limit 2` 或已知样例数量运行。
   - 确认样例输出和 Excel 已知值一致。
   - 样例通过后再全量运行。
9. 用户要求清理本次任务时，清理临时探测产物。
   - 删除探测脚本和探测输出目录。
   - 不要删除用户 Excel 文件，除非用户明确要求。
   - 不要处理无关脏工作区改动。

## 沟通要求

- 使用简体中文。
- 区分已确认页面事实和推测。
- 全量运行前，用非技术语言说明准确抽取规则。
- 结束时说明处理总数、成功数、失败数。
- 不要为了顺着用户的话跳过验证；用户催促时也要先说明当前还缺哪一步事实。
- 用户给了“基本可用”的脚本时，要明确回复：先按它复现路径，再参考成熟 app 改成稳定批量实现。
- 用户纠正条件时，要承认旧条件不完整，并说明会重新小样本验证后再全量覆盖。

## 禁止事项

- 页面有表格行列结构时，不要硬编码结果名称。
- 样例没验证通过前，不要批量写回。
- 能读取 HTML 或表格单元格时，不要只靠截图判断。
- 用户已经替换需求时，不要为旧需求保留兼容逻辑。
- 不要把“页面看起来能点”当成“批量逻辑已经可靠”。
- 不要因为用户给了一个可运行片段，就忽略仓库已有成熟实现。
- 不要把两个样例里的固定结果名称硬编码成查询全集。
- 不要在不知道表格列含义时直接写批量脚本。
