# 校对之王新增位置复核 Plan

## 1. 需求文档

校对之王在显示“参考新增”时，不能只告诉使用者“我的手册无对应原文”。使用者需要看到新增内容前后两侧共同存在的原文，并在我的手册原稿中定位到这个插入位置，从而直接判断参考手册在哪段共同内容之间增加了要求。

顶部两侧都要清楚标出新增位置附近的共同上下文：我的手册显示共同前文、插入位置和共同后文；参考手册显示共同前文、新增原文和共同后文。下方 Word/PDF 原稿同步定位；Word 必须滚到目标并产生可见高亮。

左侧修订待办使用独立滚动区。右侧原稿区适当加高，在常见桌面宽度下尽量完整显示一页 PDF。agent 不再代替 owner 启动本地服务或打开浏览器。

## 2. 当前事实

- 用户已在真实浏览器确认：参考新增事件的我的手册 Word 原稿没有定位和高亮，左侧修订待办没有形成独立滚动条。
- `reference-added` 事件当前 `myUnitIds` 为空，页面没有可用于我的手册定位的目标；这不是 Word 文件大小或渲染性能问题。
- 顺序对齐阶段已经拥有每个差异区间前后的匹配锚点，但最终 `RevisionEvent` 没有保留这些锚点。
- Word 视图已经支持根据单元 ID 在自身容器内计算滚动位置并添加 `.word-unit-focus`；有目标 ID 的普通事件可以沿用该能力。
- 工作区当前没有明确高度，左侧事件列表的内容可以继续撑高整个网格，因此 `overflow-y: auto` 不一定形成独立滚动。
- 原稿区当前高度为 `clamp(560px, 68vh, 760px)`，不足以在常见桌面宽度下完整容纳一页 PDF。
- 当前工作树还包含上一轮未提交的工程加固和校对之王改动，不回滚这些改动。

## 3. 失败测试

- 参考新增事件没有保留新增区间前后可靠共同锚点时失败。
- 参考删除事件缺少参考手册侧插入位置锚点时失败。
- 页面仍用空 `myUnitIds`/`referenceUnitIds` 定位缺失一侧原稿时失败。
- 顶部缺失一侧只显示“无对应原文”，没有共同前文、插入位置和共同后文时失败。
- 左侧事件列表不能在固定工作区内独立滚动时失败。
- Word 定位目标已存在但高亮不可见或仍不能滚到容器中部时失败。

## 4. 目标

- 新增和删除事件携带前后共同锚点，锚点来自现有顺序匹配结果，不做业务特判。
- 参考新增时，我的手册顶部显示插入位置上下文，下方 Word/PDF 定位到共同锚点。
- 参考手册顶部同时标出相同共同上下文和新增原文。
- 参考删除沿用对称逻辑，参考手册缺失一侧也能定位删除位置。
- 左侧修订待办始终拥有独立纵向滚动条。
- 原稿阅读区加高，Word 保持无横向滚动，PDF 尽量显示完整一页。
- spec、类型定义、实现和测试同步。

## 5. 不做范围

- 不修改切片、搜索召回、相似度阈值和事件分类规则。
- 不根据部门、章节、机型、岗位或样例文本特判位置。
- 不给 PDF 增加页内文字高亮。
- 不引入第三方滚动或 sidebar 组件。
- 不启动浏览器、本地服务器或 Playwright 测试。

## 6. 设计

顺序对齐完成后，为每个新增或删除事件寻找其原文区间之前和之后最近的可靠完全一致匹配。每个上下文锚点同时保存我的手册和参考手册的切片文本、单元 ID、页码及段落位置，形成成对证据。没有前锚点或后锚点时保留能确认的一侧，不虚构位置。

页面渲染新增事件时使用同一组锚点：缺失一侧显示“共同前文 + 插入位置 + 共同后文”，存在新增原文的一侧显示“共同前文 + 新增原文 + 共同后文”。共同锚点使用独立颜色，新增/删除正文继续使用原有差异颜色。

原稿定位优先使用事件自身单元；事件某侧没有原文单元时，改用该侧上下文锚点单元。Word 高亮锚点单元并滚动到第一个锚点；PDF 显示第一个锚点所在原页。

工作区使用明确响应式高度，左侧面板和右侧复核面板共享该高度。右侧顶部信息区固定，原稿区占剩余高度；左侧事件导航在剩余空间内独立滚动。窄屏恢复自然文档流。

## 9. 收口

本轮新增位置复核已完成：

- 根因确认并修复：参考新增事件原先在我的手册侧没有单元 ID，因此 Word 没有定位目标；不是性能问题。
- 新增和删除事件现在保留差异区间前后最近的完全一致匹配，形成同时指向两本手册的共同锚点；文档边界只保留真实存在的锚点。
- 顶部两侧同步显示并标记共同前文、差异原文和共同后文；缺失一侧明确显示插入或删除位置。

- 切片、搜索召回、相似度、事件分类和排序规则均未修改，也未增加业务关键词或样例特判。
