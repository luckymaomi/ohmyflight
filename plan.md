# 三本飞行手册阅读 Skills Plan

## 1. 需求文档

在 ohmyflight 的 `.agents/skills/` 中新增三份详细 Skill，分别指导 AI 和人类阅读《飞行技术管理手册》《飞行人员训练大纲》和《运行手册》。每份 Skill 要说明手册解决的问题、逐章业务重点、适用问题、带完整上下文的关键规则、三本手册之间的关系，以及收到新版本后如何完整阅读和识别新增、删除、修改。未预先熟悉手册结构的使用者也应能独立定位问题。

原始 Word/PDF 手册不得复制、提交或构建发布。Skill 只保存阅读地图和经核实的业务导航信息。

## 2. 当前事实

- 三本 DOCX 位于 owner 提供的本地目录，仅用于只读研究。
- 三份文件内部主文档类型为启用宏的 Word 文档，标准 `python-docx.Document()` 会拒绝；通过只读 OOXML 文档部件可以读取段落、标题和表格。
- 《飞行人员训练大纲》包含 50 个一级章节，主要按训练类型组织。
- 《飞行技术管理手册》包含 10 个正文一级章节和 11 个专题附录，主要覆盖训练、检查、人员资格、聘任、档案和标准管理。
- 《运行手册》包含 17 个正文一级章节，覆盖运行政策、机组、飞行、签派、适航、安保、货运、通信、性能、应急和特殊运行。
- 关键门槛同时存在于普通段落和 Word 表格，完整阅读不能只提取段落。
- 现有 `compare-manuals` Skill 能提供通用版本差异证据，不内置具体手册特判。
- `.agents/skills/` 会被构建到开发者页；owner 已确认阅读指南允许进入公开仓库，原始手册不允许。
- 开发者页当前用 `<pre><code>` 显示 Skill 原文，没有 Markdown 渲染。

## 3. 失败测试

- 任一 Skill 缺少有效 frontmatter、逐章地图、跨手册边界或版本复核流程时失败。
- 关键数字没有说明适用对象、阶段、前提、动作、结果或未完成后果，导致规则无法独立理解时失败。
- 开发者页未按运行手册、训练大纲、技术管理手册顺序置顶，或仍以代码框显示正文时失败。
- Skill 把当前阅读地图当作永远不变的手册事实，或允许收到新版本后只抽样阅读时失败。
- Skill 写入原始手册文件、绝对路径或大段原文时失败。
- 三份 Skill 不能通过 `quick_validate.py` 时失败。
- 构建后的 Skills 索引未包含三份 Skill，或全量验证失败时失败。

## 4. 目标

- 新增三份独立 Agent Skill，名称和触发描述清楚。
- 每份 Skill 能让 AI 快速判断该查哪一章，并要求最终结论回到用户提供的现行手册原文。
- 新版本复核必须覆盖完整正文、表格、附录和留空章节，并区分新增、删除、修改和待确认。
- 文档只包含通用阅读导航和已核实摘要，不包含原始手册文件。
- 开发者页使用 marked 渲染 Skill Markdown，同时保留完整源文复制。

## 5. 不做范围

- 本阶段不修改开发者页 Markdown 渲染。
- 不把手册正文、图片、PDF、DOCX或提取缓存加入仓库。
- 不把摘要当作合规结论或现行手册替代品。
- 不为具体版本文件名、部门名称变化或示例句子编写匹配特判。

## 6. 设计

每份 Skill 使用相同的信息结构：手册定位、证据规则、章节地图、业务问题读取路径、带上下文的关键规则、跨手册边界、新版本完整复核流程、输出要求和禁止事项。正文不设机械行数上限：资格链、训练链、运行限制和差异复核规则按业务需要完整展开，纯目录性内容只保留索引。删除身份化措辞、常识性解释和不提供操作路径的空泛提醒。

章节地图保存稳定的“去哪里看、关注什么”导航；数字只作为搜索锚点和复核主题，回答具体业务问题时必须重新读取用户提供的现行原文。新版本差异使用 `compare-manuals` 提取证据，再由 AI 按章节完整复核，不能直接以相似度定性。
