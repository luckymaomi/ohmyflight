# 校对之王与审计之王项目恢复闭环 Plan

## 1. 需求文档

用户要解决的实际问题：校对之王的大量修订事件需要按目录快速定位、逐项作出人工决定、按选择导出，并能携带两本原手册和工作状态换电脑继续；审计之王的检查项工作簿、PDF 工作区和原始文件当前分散保存，也需要一个完整可恢复的项目包。

谁会使用：逐章复核手册修订事件的修订人员，以及从检查项检索手册依据、整理审计依据和 PDF 页证据的审计人员。

用户完成任务时应该看到什么体验：校对之王先按大类和搜索形成完整事件流，章节树只展开小节并把事件流滚动到目标起点；每个事件可明确标为待处理、纳入报告或不纳入，导出和项目恢复保留人工决定。两个工具都能一次导出包含原始文件、结构化状态和可读 Excel 的项目 ZIP，并能在另一个浏览器完整导入继续工作。

当前范围包含：校对之王目录树、滚动联动、人工决定、全部/纳入事件 Excel 与 Word 导出、Word 空白对比说明、项目 ZIP 导入导出；审计之王检查单、手册、检查项、证据、PDF 工作区和 PDF 原文件的项目 ZIP 导入导出；共享 ZIP、哈希、下载和结构校验底层；两工具 spec、用户手册和测试。

当前范围不包含：在线服务器、账户、云同步、多人实时协作、OCR、AI 业务结论、校对对齐算法调整、审计搜索和 PDF 定位算法调整。

业务上怎样算完成：两个项目包都可在无外部文件补传的情况下恢复业务主对象、原始文件和人工工作状态；校对章节树不改变事件集合或事件数量；导出选择与页面人工决定一致；失败导入不破坏当前页面项目。

## 2. 当前事实

已确认代码事实：校对之王核心输出 `RevisionEvent`，大类和搜索生成完整事件流，章节树只提供小节滚动起点；人工决定独立于算法事件保存。`LocalManual` 保留源 `File`，手册 ID 由角色与文件名稳定生成。审计之王核心主对象仍是 `AuditKingCheckItem`，普通手册和 PDF 工作区文档均保留源文件；两个工具已接入独立项目包。

已确认测试事实：校对之王测试覆盖顺序对齐、导航、人工决定、文档视图、报告和项目包；审计之王测试覆盖检查项工作簿、来源恢复、搜索、PDF 工作区、证据导出、视图和项目包；共享归档测试覆盖工具类型与哈希校验。全量验证为 Vitest 56 个文件 201 项、Python 7 项通过。

已确认文档/spec 事实：`history.md` 明确要求结构信息只能帮助阅读和导出，不能覆盖校对全文顺序证据；审计之王检查项是唯一主对象，搜索命中、人工证据和 PDF 槽位是同一证据链的不同阶段；当前 spec 和用户手册已区分普通报告、单项工作区与完整项目 ZIP。

已确认配置或入口事实：项目是静态本地浏览器工具；两个页面均加载本地 JSZip 3.10.1 和共享归档脚本；构建会把 `src/` TypeScript 分别输出为浏览器脚本。

已确认命令输出：工作区只有当前任务相关改动；用户提供的真实校对输入位于桌面，方向为南货航 R03-01 作为我的手册、B777 R11-02 作为参考手册。

当前已有能力：校对之王可读取两本 Word/PDF、后台对齐、按目录滚动、维护人工决定、双侧定位、按全部或纳入范围导出 Excel/Word，并完整保存恢复项目；审计之王可读取检查单和多本手册、维护检查项证据链、保存检查项工作簿和 PDF 工作区、导出 PDF 证据 ZIP，并完整保存恢复项目。

当前缺口：无已知功能缺口。最终视觉主观确认由 owner 完成。

仍不明确：无。owner 已授权按最佳工程判断完成，不再等待逐项确认。

## 3. 失败测试

自动测试：校对导航测试要求章节树只提供事件起点、当前事件流始终覆盖大类与搜索结果；人工决定测试要求三态可统计、批量更新、筛选和恢复；报告测试要求 Word 对比说明为空、全部与纳入子集不丢事件、Excel记录人工决定；校对项目包测试要求包含双源文件、比较快照、人工状态和 Excel，并能校验工具类型、哈希和必需项。审计项目包测试要求包含检查单、普通手册、PDF 工作区文件、检查项证据链、槽位和工作簿，并能完整恢复。

类型检查：新增项目状态、归档清单、恢复结果、人工决定和稳定文件元数据必须有明确参数与返回结构，`npm.cmd run typecheck` 不得新增错误。

构建命令：`npm.cmd run build` 必须生成共享项目归档、两个工具项目模块和现有全部浏览器脚本。

静态检查：校对对齐四个核心模块无行为改动；页面脚本加载顺序完整；两个 ZIP 导入入口只接受 `.zip`；Word 表头保持五列且对比说明数据为空。

真实用户路径演练：用用户提供的两本真实 PDF 完成校对，记录四类事件数；逐章跳转、改变多个事件人工决定、导出纳入 Excel/Word、导出项目 ZIP；新页面导入 ZIP 后核对源文件、事件数、决定数、目录跳转、PDF 页面和再次导出。审计之王使用合成但真实格式的检查单、手册和 PDF 工作区项目执行导出、清空、导入和证据关系恢复。

失败判定：章节点击改变校对事件总集合、人工决定丢失、项目导入依赖外部补传文件、错误 ZIP 部分覆盖当前状态、Word 对比说明非空、报告行数与导出范围不一致、项目包工具类型可串用、核心算法结果变化、构建或全量测试失败，均判定未完成。

## 4. 目标

用户路径完成到：上传或导入项目 -> 查看完整结果 -> 目录跳转 -> 人工决定 -> 全部或纳入导出 -> 保存项目 -> 换页面导入继续。

代码主链路接到：共享归档底层只处理 ZIP、文件哈希、文件提取和下载；校对项目模块消费 `RevisionEvent` 与 `LocalManual`；审计项目模块消费 `AuditKingCheckItem`、普通文档和 PDF 工作区；各自恢复业务状态。

状态和记录落到：校对人工决定按事件 ID 保存为待处理、纳入报告、不纳入；审计保存检查项、候选证据、审计依据、PDF 槽位和视图状态；原始文件以二进制写入各自 ZIP；Excel 是可读镜像，不是唯一状态源。

输出呈现到：校对目录树为章节折叠、小节叶节点和完整事件流；导出菜单区分全部和纳入范围；Word 对比说明留空；项目栏显示导入、保存和未保存状态。审计页面增加清晰的导入项目、保存项目入口，不替换现有单项工作簿和 PDF 证据出口。

测试和文档同步到：`tests/tool/proof-king/`、`tests/tool/audit-king/`、两工具开发规格和用户手册、根计划同步当前事实。

完成判定：局部测试、真实恢复演练、构建、类型检查和全量测试全部通过；项目 ZIP 反向读取闭合；用户可通过本地预览地址人工检查页面。

## 5. 不做范围

不做：通用工作流平台、跨工具项目互导、服务器上传、账户系统和协同编辑。

不保留：把校对章节或小节作为事件过滤器的重复状态；把 Excel 或 Word 冒充完整项目恢复格式的隐含入口。

不兼容：不为尚不存在的旧项目 ZIP 设计兼容层；项目清单版本不支持时明确拒绝。

不处理：扫描 PDF、图片 PDF、旧 `.doc`、损坏 ZIP 和缺少文字层的输入。

## 6. 设计

主链路：源文件 -> 业务读取/算法 -> 唯一主对象 -> 人工状态 -> 报告；保存项目时把源文件、结构化状态和可读报告写入 ZIP；导入时先完整解包、校验、读取和重建候选状态，全部成功后一次替换页面。

输入 -> 判断 -> 状态 -> 执行 -> 输出 -> 记录：识别项目工具类型和版本；验证清单、文件路径、哈希和状态结构；构造临时业务状态；完成读取和关系重绑；一次提交到页面；显示恢复数量与未保存状态。

模块边界：共享模块不认识修订事件、检查项或 PDF 槽位；校对导航模型只计算大类、搜索、章节树和事件起点；校对人工决定模型独立于算法事件；两个项目模块各自负责清单和业务快照。

文件职责：新增共享 `src/tool/project-archive.ts`；校对新增人工决定与项目模块并调整导航视图、工作区和报告；审计新增项目模块并让读取边界保留源文件；页面只接线和呈现。

状态归属：校对 `RevisionEvent` 不可变，人工决定另存；审计 `AuditKingCheckItem` 继续作为唯一证据链对象；搜索命中和校对目录均为派生视图；ZIP 内 Excel 和 Word 均为派生输出。

数据或事件流：校对大类与搜索生成完整 `visibleEvents`，章节树基于同一集合生成锚点；点击小节设置虚拟列表滚动位置并选中起点事件。项目导入读取源文件并恢复同包快照，校验后重建视图。审计导入先读取检查单、手册与 PDF，再恢复检查项和槽位并重算搜索。

错误、恢复、中断、重试边界：不设置人为文件数量或大小上限；压缩、解压、读取显示阶段进度；任何必需文件、哈希或状态失败都保留当前项目；导出失败不清除脏状态；成功保存项目后清除未保存提示。

与现有架构的关系：继续使用本地 JSZip、SheetJS、docx、PDF.js 和 Web Worker；不引入框架或服务端；校对核心对齐模块不修改；审计现有工作簿、工作区 JSON 和证据 PDF 导出继续保留。

测试和文档影响：项目归档需要纯模型单测、错误包测试、浏览器真实导入恢复；报告范围和人工状态需要新增单测；两工具用户手册需明确项目 ZIP 与普通报告的区别。
