# ohmyflight Agent 工作规约

本文件面向在本仓库内工作的模型和 agent。它不是宣传文档，而是开发约束。

## 最高原则：事实高于一切

所有与用户的交互、所有与用户的交流、所有发言、所有判断、所有计划、所有代码修改，都必须基于代码中的客观实际，以及用户呈现出来的客观文本、截图、HTML、数据、运行结果和明确需求。

如果 agent 造假，把不存在的功能说成存在，把没有验证的猜测说成事实，把未来计划说成当前现实，把幻想、推测、愿望、方案草稿、未完成实现当作已经发生的事实，那么本次工作将失去意义。本项目的代码将失去意义，开发行为将失去意义，所有后续判断都会被污染。

因此，任何时候都必须先区分：

- 已经从代码或运行结果确认的事实
- 用户明确提供的事实
- 尚未验证的推测
- 计划做但还没有做的事情
- 已完成并通过验证的事情

不能把这些东西混在一起。事实不清楚时，先排查；无法排查时，明确说明不确定。不要用自信语气替代证据。

## 沟通方式

- 始终使用简体中文和项目 owner 交流。
- 回复要简短、直接、可执行。
- 不写空泛长文，不做仪式化总结。
- owner 关心的是运行事实、产品效果、风险点和验证结果；除非明确要求，不要展开代码讲解。

## 事实纪律

- 所有判断、建议和代码修改都必须基于客观事实。
- 客观事实包括：仓库代码、规格文档、测试、命令输出、用户提供的文本/截图/HTML、可复现的运行结果。
- 不要编造不存在的功能、数据、接口、页面行为、测试结果或业务规则。
- 如果事实不清楚，先排查再动手；影响实现方向的问题不能靠猜。
- 脱离事实的开发没有意义。

## 项目重点

- 培训模块是本项目最重要的模块。
- 涉及培训有效期、培训预排、培训冲突、培训规则、培训导出的改动，必须格外谨慎。
- 培训规则以已经确认的业务事实、`spec/` 文档和测试为准，不以模型直觉为准。

## 开发纪律

- 修改前先读相关代码、规格和测试。
- 优先沿用现有项目结构、工具函数和实现风格。
- 改动范围要紧贴当前任务，不顺手重构无关模块。
- 不要回滚用户已有改动，除非 owner 明确要求。
- 遇到脏工作区时，只处理和任务相关的文件。

## 规格与测试

- `spec/` 是用户侧功能事实和重要业务规则的记录位置。
- 代码、测试、文档必须同步。任何一方描述了旧事实，都会污染后续开发。
- 行为变化后，如果旧规格或旧文档已经不真实，必须同步更新。
- 解析规则、业务规则、回归问题、共享逻辑变化时，必须补充或更新测试。
- 本地一次性工具和稳定小工具可以主要依赖 smoke test、语法检查和入口覆盖；但核心业务规则必须有明确测试保护。
- 仓库发生代码改动后，收尾前必须运行全量测试；如果无法运行，必须明确说明原因和风险。
- 不能只凭解释声称完成，必须说明验证结果。

## TDD 测试驱动开发原则

本项目坚持测试驱动开发。

测试的目标不是覆盖一切，也不是为了覆盖 UI、目录形状、文件拆分、样式细节或无关紧要的页面结构。测试的目标是保护业务核心规则不被破坏。

必须重点保护：

- 培训有效期规则
- 培训预排规则
- 培训计划核对规则
- 日期窗口和到期判断
- Excel 解析和导出结构中的核心字段
- 文本解析、姓名匹配、员工号提取、锁班类型识别等核心解析逻辑
- 曾经出过问题的回归场景

不追求无意义的测试覆盖率。不为了显得“工程化”而写脆弱测试。不用测试锁死不重要的 UI 结构、文件拆分方式或临时实现细节。

修复回归、修改业务规则、调整解析逻辑、改变共享行为时，优先把失败场景写成测试，再修改实现。不能先靠感觉改代码，再用解释证明自己是对的。

可接受的开发顺序是：

1. 确认事实和失败场景
2. 写出失败测试或更新现有测试
3. 修改实现
4. 运行相关测试
5. 运行全量测试
6. 同步文档和规格

对一次性本地小工具，可以不为所有 UI 细节补测试；但核心解析、业务规则、日期规则、导出结构、培训规则和曾经出过问题的回归点，必须优先用测试保护。

## 完成标准

每次开发完成前，必须检查三件事是否一致：

- 代码是否实现了当前确认的事实
- 测试是否保护了关键行为
- 文档或规格是否还在描述真实状态

如果三者不一致，任务不能算完成。应先修正不一致，再运行全量测试，再收尾。

## 运行证据

- 用户给了原始输入时，优先用原始输入复现。
- 表格、文本解析、日期和姓名问题，要查看中间值，不要只看最终现象。
- 浏览器自动化问题要分清：输入解析、DOM 选中值、页面显示值、实际提交值。
- 内网系统无法访问时，以用户提供的 HTML、截图、日志和本地代码为事实来源。

## 收尾

- 最终回复保持简洁。
- 说明改了什么、验证了什么、是否还有风险。
- 不粘贴大段日志、diff 或源码，除非 owner 明确要求。
