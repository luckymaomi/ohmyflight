# ohmyflight Agent 工作规约

本文件面向在本仓库内工作的模型和 agent。它不是宣传文档，而是开发约束。

## 文件读取编码

- 阅读本仓库所有文本文件时，必须按 UTF-8 读取。
- PowerShell 直接 `Get-Content` 中文文件时可能出现终端显示乱码；判断文件内容是否损坏时，必须用 UTF-8 读取确认，不要把终端乱码误判为文件已坏。

## 最高原则：事实高于一切

所有与用户的交互、所有与用户的交流、所有发言、所有判断、所有计划、所有代码修改，都必须基于代码中的客观实际，以及用户呈现出来的客观文本、截图、HTML、数据、运行结果和明确需求。

如果 agent 造假，把不存在的功能说成存在，把没有验证的猜测说成事实，把未来计划说成当前现实，把幻想、推测、愿望、方案草稿、未完成实现当作已经发生的事实，那么本次工作将失去意义。本项目的代码将失去意义，开发行为将失去意义，所有后续判断都会被污染。

因此，任何时候都必须先区分：

- 已经从代码或运行结果确认的事实
- 用户明确提供的事实
- 尚未验证的推测
- 计划做但还没有做的事情
- 已完成并通过验证的事情

不能把这些东西混在一起。事实不清楚时，先排查；无法排查时，明确说明不确定。不要用自信语气替代证据。

## 沟通方式

- 始终使用简体中文和项目 owner 交流。
- 回复要简短、直接、可执行。
- 不写空泛长文，不做仪式化总结。
- owner 关心的是运行事实、产品效果、风险点和验证结果；除非明确要求，不要展开代码讲解。

## 事实纪律

- 所有判断、建议和代码修改都必须基于客观事实。
- 客观事实包括：仓库代码、规格文档、测试、命令输出、用户提供的文本/截图/HTML、可复现的运行结果。
- 不要编造不存在的功能、数据、接口、页面行为、测试结果或业务规则。
- 如果事实不清楚，先排查再动手；影响实现方向的问题不能靠猜。
- 脱离事实的开发没有意义。
- 所有文档只写当前事实，不写历史过程，不写未来计划，不写未经验证的推测。

## 项目重点

- 培训模块是本项目最重要的模块。
- 涉及培训有效期、培训预排、培训冲突、培训规则、培训导出的改动，必须格外谨慎。
- 培训规则以已经确认的业务事实、`spec/` 文档和测试为准，不以模型直觉为准。

## 开发纪律

- 修改前先读相关代码、规格和测试。
- 优先沿用现有项目结构、工具函数和实现风格。
- 改动范围要紧贴当前任务，不顺手重构无关模块。
- 需求明确改为新事实时，只保留新事实；禁止为了旧实现、旧字段、旧入口或旧行为保留兼容层。
- 不要回滚用户已有改动，除非 owner 明确要求。
- 遇到脏工作区时，只处理和任务相关的文件。

## 项目级 Skills

- 项目级 skill 放在 `.agents/skills/`。
- 任务命中项目级 skill 描述时，必须先阅读对应 `SKILL.md`，再执行任务。
- 涉及 Playwright 临时脚本、内网页面查询、Excel 批量查询时，必须先看 `.agents/skills/playwright-probe/SKILL.md`。
- 用户提供的 Playwright 录制脚本是业务需求和业务路径，不代表成熟稳定的最终实现。
- 项目级 skill 是本仓库内的具体工作规程；如果和 owner 当前明确需求冲突，以 owner 当前需求为准，但必须说明冲突点和处理方式。

## 规格与测试

- `spec/` 是用户侧功能事实和重要业务规则的记录位置。
- 代码、测试、文档必须同步。任何一方描述了旧事实，都会污染后续开发。
- 行为变化后，如果旧规格或旧文档已经不真实，必须同步更新。
- 解析规则、业务规则、回归问题、共享逻辑变化时，必须补充或更新测试。
- 本地一次性工具和稳定小工具可以主要依赖 smoke test、语法检查和入口覆盖；但核心业务规则必须有明确测试保护。
- 测试只写符合当前事实的正向行为和当前规则，不写用于拦截旧实现、旧字段、旧入口或旧行为的负向兼容测试。
- 仓库发生代码改动后，收尾前必须运行全量测试；如果无法运行，必须明确说明原因和风险。
- 不能只凭解释声称完成，必须说明验证结果。

## TDD 测试驱动开发原则

本项目坚持测试驱动开发。

测试的目标不是覆盖一切，也不是为了覆盖 UI、目录形状、文件拆分、样式细节或无关紧要的页面结构。测试的目标是保护业务核心规则不被破坏。

必须重点保护：

- 培训有效期规则
- 培训预排规则
- 培训计划核对规则
- 日期窗口和到期判断
- Excel 解析和导出结构中的核心字段
- 文本解析、姓名匹配、员工号提取、锁班类型识别等核心解析逻辑
- 曾经出过问题的回归场景

不追求无意义的测试覆盖率。不为了显得“工程化”而写脆弱测试。不用测试锁死不重要的 UI 结构、文件拆分方式或临时实现细节。

修复回归、修改业务规则、调整解析逻辑、改变共享行为时，优先把失败场景写成测试，再修改实现。不能先靠感觉改代码，再用解释证明自己是对的。

可接受的开发顺序是：

1. 确认事实和失败场景
2. 写出失败测试或更新现有测试
3. 修改实现
4. 运行相关测试
5. 运行全量测试
6. 同步文档和规格

对一次性本地小工具，可以不为所有 UI 细节补测试；但核心解析、业务规则、日期规则、导出结构、培训规则和曾经出过问题的回归点，必须优先用测试保护。

## 完成标准

每次开发完成前，必须检查三件事是否一致：

- 代码是否实现了当前确认的事实
- 测试是否保护了关键行为
- 文档或规格是否还在描述真实状态

如果三者不一致，任务不能算完成。应先修正不一致，再运行全量测试，再收尾。

## 运行证据

- 用户给了原始输入时，优先用原始输入复现。
- 表格、文本解析、日期和姓名问题，要查看中间值，不要只看最终现象。
- 浏览器自动化问题要分清：输入解析、DOM 选中值、页面显示值、实际提交值。
- 内网系统无法访问时，以用户提供的 HTML、截图、日志和本地代码为事实来源。
- 前端视觉效果由 owner 人工检查；agent 只做构建、测试和静态页面连通性验证，不做截图级视觉验收。

## 收尾

- 最终回复保持简洁。
- 说明改了什么、验证了什么、是否还有风险。
- 不粘贴大段日志、diff 或源码，除非 owner 明确要求。

## 提交与推送

- 提交信息必须使用简体中文。
- 提交信息必须详细写明本次提交实际包含的所有改动，不要用笼统描述替代事实。
- 提交前必须确认暂存区只包含本次任务相关文件；真实业务数据、临时本地文件和与任务无关的脏工作区改动不要混入提交。

## 仓库事实与工作入口

以下内容基于当前仓库结构和命令确认。

- 本项目是静态浏览器工具集合，源码主要在 `src/`，静态页面、第三方库和模板主要在 `public/`，构建产物输出到 `dist/`。
- 构建入口是 `npm run build`，实际执行 `tsx scripts/build.mts`：先打包源码压缩包，再清空并复制 `public/` 到 `dist/`，最后把 `src/**/*.ts` 转译为对应的 `dist/**/*.js`。
- 测试入口以 `package.json` 为准：`npm test` 会先构建再运行 `vitest run`；`npm run typecheck` 只做 TypeScript 类型检查。
- `spec/` 是用户侧功能事实和业务规则文档。
- 培训相关核心实现集中在 `src/tool/app/training-workbench/`、`src/tool/app/training-tools-suite/`、对应 `public/tool/app/...` 页面、`spec/app/...` 文档和 `tests/tool/...` 测试。
- 首页是 `public/index.html` 和 `src/home/*`；工具入口页是 `public/tool/index.html`、`public/tool/index.css`、`src/tool/tools-render.ts`、`src/tool/tool-mode-toggle.ts`、`src/tool/tools-data.ts`。
- 当前仓库包含 Bootstrap 5.3.3 的本地文件：`public/libs/bootstrap.min.css` 和 `public/libs/bootstrap.bundle.min.js`。大量工具页直接引用这些本地 Bootstrap 文件。
- 当前仓库已有 `public/libs/versions.json` 记录部分本地依赖版本。
- PowerShell 直接 `Get-Content` 中文文件时可能出现终端显示乱码；判断文件内容是否损坏时，优先用 Node 以 UTF-8 读取确认，不要把终端乱码误判为文件已坏。
