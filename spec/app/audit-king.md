# 审计之王

## 用途

用于把检查单中的检查项与公司手册依据组织成可复核的证据链，并把最终审计依据定位到文字型 PDF 页面。

## 主对象

检查项是唯一主对象。每个检查项包含：

- 检查编号：人工维护，可为空。
- 检查项名称：人工维护，可为空。
- 关键词：人工维护，一个检查项一个关键词，可为空。
- 检查单来源：可选的原文位置。
- 手册证据：从搜索命中或手册全文选中保存，可有多条。
- 审计依据：人工新增或从手册证据采纳，可有多条。

编号、名称和关键词互不推断。导入缺少某个字段时保持空值。

## 工作流

1. 上传标准 `.docx` 检查单。
2. 上传一本或多本标准 `.docx` 或带文字层的 `.pdf` 公司手册。
3. 创建检查项，填写编号、名称和关键词。
4. 可在检查单原文选中文字创建检查项，或更新当前检查项的来源位置。
5. 系统使用启用的检查项关键词检索启用的公司手册。
6. 用户把命中摘要或全文选区保存为当前检查项的手册证据。
7. 用户确认手册证据后点击“采纳为审计依据”。采纳只追加一条依据，不删除候选证据，不覆盖已有依据；同一证据不会重复采纳。
8. 审计依据支持人工新增、编辑和删除。
9. 从审计依据生成 PDF 证据槽位，自动识别候选连续页，人工确认后导出。

检查项的新增、删除、排序、编号和名称修改直接作用于同一对象，检查项列表与审计篮子不存在独立副本。

## 搜索规则

- 原文完全包含关键词时记为精确命中。
- 宽松命中对文本执行 NFKC、小写化，并忽略 Unicode 空白、标点和符号。
- 宽松命中必须映射回原文起止坐标，页面展示和证据保存始终使用原文。
- 本地二元 gram 索引用于缩小候选文本块，最终命中仍由原文包含关系确认。
- 搜索不做语义判断，不判断依据是否充分，不包含业务关键词、部门名、章节名或手册名特判。
- PDF 手册命中保留页码；Word 手册命中保留段落位置。

## 检查项工作簿

当前格式包含且必须包含三个 sheet：

- `检查项`：序号、检查编号、检查项名称、关键词、启用、颜色和检查单来源定位字段。
- `手册证据`：按检查项序号保存多条候选证据、手册名、段落、PDF 页码、全文坐标、原文和上下文。
- `审计依据`：按检查项序号保存多条依据、备注和可选来源证据 ID。

导入当前工作簿会替换页面中的检查项集合。旧关键词工作簿和旧审计篮子不是当前导入格式，不做兼容读取。

审计篮子可另行导出为传统结果表，字段为条款名称、依据序号、依据内容、备注，用于后续工作交付；该结果表不能反向导入为检查项。

## 文档读取

- Word 使用本地 Mammoth 读取原始文字并按段落形成搜索块。
- PDF 使用本地 PDF.js 读取文字层，按页面和视觉行形成搜索块并记录页码。
- 扫描件、图片 PDF 和 OCR 不在当前范围。
- 手册只在浏览器本地处理，不上传服务器。

## PDF 证据工作区

- 从当前检查项的审计依据生成槽位，不保存第二份审计篮子状态。
- 每本 PDF 生成 1 页、2 页、3 页连续窗口；同一轮识别只构建和归一化一次窗口。
- 依据按行归一化，长行使用约 26 字、步长约 20 字的滑动切片。
- 候选按片段加权覆盖率、片段顺序和窗口密度评分。
- 覆盖不足或有效片段少于 2 个保持未命中；算法只提供候选，不替代人工确认。
- 业务忽略行、弱片段或跳过规则默认均为空，只能通过 `AuditKing.PdfLocatorHooks` 外部注入。
- 支持工作区 JSON、页码调整、前后扩页、预览和 PDF 导出。

## 文件夹 Python

页面顶部提供编号范围输入和 Python 导出。输入默认为空，placeholder 只提示 `1.1-1.61` 格式；不显示文件夹预览。生成脚本在运行目录逐个创建编号文件夹。

## 完成边界

- 页面不自动生成检查项编号、名称或关键词。
- 页面不自动采纳手册证据。
- 页面不把搜索命中当成审计结论。
- 页面不内置具体公司、部门、手册、章节、标题或业务词池。
