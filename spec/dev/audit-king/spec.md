# 审计之王开发规格

## 产品定位

`audit-king` 把检查单条目、公司手册候选证据、人工确认的审计依据和最终 PDF 页证据组织在同一条可复核链路中。工具只提供文本检索、来源定位和证据整理，不自动判断合规性或依据是否充分。

## 核心模型

检查项是唯一主对象，每项独立保存人工维护的检查编号、检查项名称、单个关键词、启用状态、颜色、检查单来源、多条手册证据和多条审计依据。编号、名称和关键词互不推断；缺失字段保持空值。

候选手册证据与审计依据是两个阶段：搜索命中或全文选区先保存为候选证据；“采纳为审计依据”只追加依据，不删除候选、不覆盖旧依据，同一来源不得重复采纳。人工依据可新增、编辑、删除。

## 文档输入

- 检查单支持标准 `.docx`，使用 Mammoth 在浏览器本地读取并保留稳定段落坐标。
- 手册支持标准 `.docx` 与带文字层的 `.pdf`；PDF.js 按页读取文字层并保留页码。
- 扫描 PDF、图片 PDF、OCR 和旧 `.doc` 不在范围。
- 文档 ID 和块 ID必须稳定，不得依赖当前时间，否则工作簿恢复后来源会漂移。

## 搜索契约

- 精确命中要求原文完整包含关键词。
- 宽松命中使用 NFKC、小写化，并忽略 Unicode 空白、标点和符号；命中坐标必须反向映射到原文。
- FlexSearch 二元 gram 索引只负责候选召回，最终仍由原文包含关系确认。
- 不做语义相似度结论，不内置公司、部门、手册、章节或业务词特判。
- 结果可按关键词和启用手册筛选，筛选不改变证据归属。

## 检查项工作簿

可导入格式必须同时包含 `检查项`、`手册证据`、`审计依据` 三个 sheet，并按检查项序号关联。导入会替换页面当前检查项集合；旧关键词表或传统审计篮子结果表不兼容。传统审计篮子仍可单独导出供交付，但不能反向导入。

来源恢复优先使用文档名、块索引和坐标；文本移动时可用上下文唯一匹配恢复。存在多个相同候选时必须保持未定位，不猜测来源。

## PDF 证据工作区

- 从当前审计依据即时生成槽位，不维护第二份审计篮子。
- 每本 PDF 构建连续 1、2、3 页候选窗口；依据按有意义片段、顺序和窗口密度评分。
- 覆盖不足或有效片段少于两个时保持未命中；自动结果只是人工确认候选。
- 自动命中默认可向前后各扩一页但不得越界，用户可改页码、预览、保存工作区 JSON 和导出 PDF。
- 业务忽略行或弱片段规则默认为空，只允许通过 `AuditKing.PdfLocatorHooks` 外部注入。

## 附属能力与边界

页面可按一个完整编号范围生成独立 Python 脚本，脚本只在运行目录逐个创建编号文件夹；范围无效时拒绝生成。页面不自动生成检查项字段、不自动采纳证据、不把检索结果当审计结论。
