# ohmyflight 案例与贡献页面开发规格

## 产品定位

`sponsor-page` 是 ohmyflight 的独立案例与贡献页面，面向希望快速了解项目实际价值并参与开源协作的人。页面只展示真实运行画面、已确认的能力边界和仓库现有贡献入口，不新增工具能力，也不把宣传文案当作业务规则。

## 页面入口

- 页面路径：`public/sponsor/index.html`
- 工具首页入口由 `src/site-visibility.ts` 中的 `homepage.sponsorEntry` 控制。

## 展示案例

- 猫咪：排在首位，展示一句指令处理文件任务，并提供带“猫咪官网”前缀的官网链接，以及全局安装、项目初始化和启动命令。
- 锁班皇帝：按人员、类型和日期批量录入非生产任务，并逐条保存成功、冲突和错误结果。
- 起落数查询助手：人工登录飞行门户后，按员工号和日期区间逐人查询飞行经历或左座经历与起落数，并把结果汇总到本地 Excel。
- 自动点 OA 助手：处理所有可直接确认的“已阅”待办；审批、意见、签署和特殊事项仍由人工完成。
- 培训皇帝：读取培训总表，按现有规则展示排班风险并导出新副本。

## 视觉与数据边界

- 使用 owner 提供并确认只含样例数据的 GIF 演示，不使用虚构界面。
- 站点保存独立 GIF 副本，不修改 owner 的原始 demo 文件。
- 桌面和移动端都必须保证标题、按钮和媒体不重叠；GIF 使用固定纵横比，避免加载时页面跳动。
- 案例正文不使用编号、装饰性小字、装饰性下划线和案例跳转按钮；贡献区使用清楚的分区边界和标准文本链接。
- 页面不设置顶部导航；最前方使用“为什么是 ohmyflight？”作为独立 Hero，并在标题下展示与项目 README 一致的 Pages、TypeScript、Python、Vitest、静态本地优先和 MIT 徽标；徽标使用与 Kitty README 一致的圆角风格；Hero 不使用动态背景，并在首屏露出后续案例；动态演示只出现在案例正文。
- Kitty 的三个命令使用白底黑字控件和黑色复制按钮，复制成功后短暂显示“已复制”。
- 五个案例之后直接显示“一起建设 ohmyflight”、贡献指南、Issues、Pull requests 和贡献人员，不重复项目名称、公开状态、许可证、技术栈或仓库入口。
- 协作路径使用类似仓库文件列表的纵向结构，不使用营销卡片；贡献区使用中性灰白界面和链接蓝，不展示付款码或收款文案。
- 贡献区不使用 `OPEN SOURCE`、`COMMUNITY` 等装饰性小标签。
- 贡献人员区域显示当前发布人数，并用头像、姓名和具体贡献组成独立人员记录。
- 贡献人员名单集中维护在 `public/sponsor/contributors.js`。每项包含姓名、贡献说明、可选链接和可选头像链接；链接不限 GitHub，留空时姓名与头像不可点击，适合只记录姓名和贡献的同事。头像支持站内本地图片和 `http/https` 网络图片；未提供头像或头像加载失败时使用本地匿名 SVG。`src/site-visibility.ts` 中的 `sponsorPage.contributors` 为 `false` 或名单为空时，页面显示配置的空状态，不渲染未确认人员。
- 贡献人员标题、等待状态、空状态和发布人数模板统一维护在同一文件的 `labels` 中；人数模板使用 `{count}` 占位符。HTML 不保存这些文案。
