# 培训工具合集维护说明

## 1. 当前目录包含什么

当前目录包含 4 个功能页和 1 个说明页：

- `index.html`
  培训工具合集入口页。
- `schedule.html`
  培训预排班。
- `validity-update.html`
  培训有效期更新。
- `conflict-check.html`
  培训冲突核对。
- `rule-reference.html`
  飞行部培训规则速查、字段规范、模板下载说明。

对应脚本在 `scripts/` 目录：

- `core.js`
  通用基础能力。规则默认值、Excel 读写、表头识别、日期处理、模板导出都放在这里。
- `rule-engine.js`
  与 `super-training-test` 对齐的规则判定层，负责更新结果判定和区间预排判定。
- `validity-engine.js`
  培训有效期更新的业务引擎。负责读取“规则表 + 某个培训表”的组合文件，再和有效期表做匹配、计算和导出。
- `schedule-engine.js`
  培训预排班业务引擎。负责按预排区间筛人，并生成导出工作簿。
- `update.js`
  培训有效期更新页的页面交互与渲染。
- `schedule.js`
  培训预排班页的页面交互与渲染。
- `conflict.js`
  培训冲突核对页逻辑。
- `reference.js`
  规则速查页逻辑。

样式统一放在：

- `styles/suite.css`

## 2. 这次调整后的维护原则

这套工具保留 `training-tools-suite` 的分页面风格，但底层业务逻辑要和 `super-training-test` 保持一致。

维护时建议按下面分层处理：

- 通用字段、日期解析、Excel 读写、模板下载改动：先看 `scripts/core.js`
- 更新结果判定、预排状态判定改动：先看 `scripts/rule-engine.js`
- 培训有效期更新流程改动：改 `scripts/validity-engine.js`
- 培训预排流程改动：改 `scripts/schedule-engine.js`
- 页面交互、状态提示、表格渲染改动：改对应页面脚本

不要再把复杂业务判断直接散落到页面脚本里。

## 2.1 当前窗口规则补充

- `危险品`
  有效期按 2 年计算；复训窗口是“失效日前 3 个日历月至失效前 1 日”。
- `英语能力 / 汉语能力`
  英语能力按 `ICAO4` 有效期 3 年，汉语能力按 `ICAO5` 有效期 6 年；考试窗口都是“失效日前 3 个日历月至失效日当天”。
- `90 / 60 / 30 天内到期`
  这组统计仍然保留，按“距离本轮必须完成日期还有多少天”分层。
- `培训开始日期`
  有效期更新一律以培训开始日期作为新生效锚点；培训结束日期只用于留档和表格展示。

## 3. 当前 4 个功能的导入口径

### 3.1 培训预排班

输入：

- 规则表文件
- 有效期表文件
- 项目
- 预排开始日期
- 预排结束日期

识别规则：

- 规则表优先识别名为 `规则表` 的工作表
- 若未命名，也会按规则字段自动匹配
- 有效期表默认读取第一张工作表

关键口径：

- 现在是“区间预排”，不是单日预排
- 有窗口项目：看预排区间是否与窗口重叠
- 无窗口项目：看旧有效期是否落在本阶段内
- 输出状态与成品版一致：`已过期 / 命中窗口 / 本阶段到期 / 有效期未到窗口 / 阶段外未到期 / 缺少旧有效期`

导出结果：

- `摘要`
- `已过期`
- `命中窗口`
- `本阶段到期`
- `缺少旧有效期`
- `未命中窗口`

### 3.2 培训有效期更新

输入：

- 培训表文件
  文件内必须包含 `规则表 + 某个培训表`
- 有效期表文件

关键字段：

- 培训表至少包含：
  `员工号 / 姓名 / 项目名称 / 培训开始日期 / 培训结束日期 / 培训信息是否录入`
- 有效期表至少包含：
  `员工号 / 姓名`
  以及需要更新的项目列

关键口径：

- 只有 `培训信息是否录入 = 是` 才参与更新
- 人员匹配、窗口命中、结果分类和回写限制与 `super-training-test` 一致
- 更新结果分为：
  `已更新 / 不变 / 有效期回退 / 更新无效 / 跳过`

导出结果：

- `摘要`
- `更新预览`
- `跳过记录`
- `更新后有效期表`

其中“更新后有效期表”会高亮本次真正回写成功的有效期单元格。

### 3.3 培训冲突核对

输入：

- 1 个培训表文件

最少字段：

- `姓名 / 项目名称 / 培训开始日期 / 培训结束日期`

识别规则：

- 若导入的是整本培训工作簿，优先识别名为 `培训表` 的工作表
- 若未命名，也会按这 4 列自动匹配

关键口径：

- 现在仍按“同名人员”分组
- 只要两个培训日期区间重叠，就判定冲突
- 首尾同日也算冲突

### 3.4 飞行部培训规则速查

职责：

- 展示规则口径
- 展示项目对照
- 展示案例说明
- 展示字段规范
- 提供模板下载入口

注意：

- 这里只是说明页，不直接参与计算
- 但只要导入要求、输出结果、规则口径变了，就要同步更新这里

## 4. 哪些位置是关键源头

### 4.1 默认规则和项目别名

在 `scripts/core.js`：

- `DEFAULT_RULES`
- `PROJECT_LOOKUP`

新增项目或改别名时：

1. 先改 `DEFAULT_RULES`
2. 再检查 `rule-reference.html`
3. 再检查模板导出是否要同步

### 4.2 业务规则判定

在 `scripts/rule-engine.js`：

- `classifyUpdateJudgement`
- `evaluateUpdateResult`
- `classifyScheduleStageStatus`

只要成品版 `super-training-test` 的这些判定变化了，这里必须同步。

### 4.3 表头规范

在 `scripts/core.js`：

- `RULE_REQUIRED_HEADERS`
- `TRAINING_REQUIRED_HEADERS`
- `CONFLICT_REQUIRED_HEADERS`
- `VALIDITY_REQUIRED_HEADERS`

如果业务说“最少字段”变了，优先改这里。

### 4.4 业务引擎

在：

- `scripts/validity-engine.js`
- `scripts/schedule-engine.js`

这里负责把“读表、匹配、计算、结果归类、导出结构”串起来。页面脚本只负责 UI，不负责埋业务规则。

### 4.5 模板导出

模板不是静态文件源，而是运行时从 `scripts/core.js` 生成：

- `exportTrainingTemplate()`
- `exportValidityTemplate()`

## 5. 改完后最低限度自查

建议至少做这些：

1. 语法检查
   对改过的 JS 跑 `node --check`
2. 工具测试
   跑 `node node_modules/vitest/vitest.mjs run tests/tool/training-tools-suite`
3. 页面入口检查
   确认 `index.html` 可以打开对应页面
4. 导入检查
   用真实 Excel 验证规则表、培训表、有效期表能否被识别
5. 导出检查
   确认导出的工作表名称、日期格式、高亮效果和统计是否正确

## 6. 当前已知设计

- 有效期表仍默认读取第一张工作表，这是当前设计，不是 bug
- 冲突核对仍按“姓名”分组，如果以后要改成员工号优先，需要单独升级
- 套件版继续保留分页面入口，但底层更新和预排判定要与 `super-training-test` 同步
